# Rating Comment Style Guide

Use this guide when drafting or reviewing rating comments. The goal is to write comments that sound natural, personally reasoned, and professional without feeling templated or overly polished.

---

## 1. Core Writing Goal

Write like a careful human rater briefly explaining:

1. what the user appears to want,
2. what the result or creative appears to offer,
3. why that supports the rating.

A good comment should feel:
- specific,
- grounded in visible evidence,
- concise,
- natural in phrasing,
- appropriately confident or cautious.

---

## 2. Use Plain, Concrete Wording

Prefer ordinary language over abstract evaluation language.

### Prefer
- “The user is looking for…”
- “The page shows…”
- “The creative appears to offer…”
- “This matches the product type, but not the brand.”
- “That makes it related, but not exact.”

### Avoid
- “The result demonstrates strong alignment with the user’s expressed intent.”
- “The offering partially fulfills the core objective.”
- “This exhibits limited utility despite thematic overlap.”

### Rule
Use the simplest wording that accurately explains the judgment.

---

## 3. Be Specific About the Decisive Reason

Do not stop at generic words like “relevant,” “related,” or “not a match.” State what specifically matches or fails.

### Prefer
> The product itself matches, but the retailer does not. The user asked for Walmart and this is from Target.

### Avoid
> The result is relevant but not fully aligned with the query.

### Rule
Name the actual feature that drives the judgment:
- product type,
- brand,
- retailer,
- size,
- model,
- function,
- missing claim,
- unclear evidence,
- substitute quality.

---

## 4. Use a Natural Reasoning Flow

A reliable structure is:

> The user wants X. The result offers Y. Since Z is preserved or missing, I rated it accordingly.

### Example
> The user is looking for creatine gummies at Walmart. The result shows creatine gummies at Target. Since the product matches but the retailer changes, this is a reasonable alternative rather than an exact match.

### Rule
Use this structure when helpful, but do not force every comment into the exact same wording.

---

## 5. Vary Sentence Length and Rhythm

Comments sound more human when they do not all have the same sentence shape.

### Prefer
> The user wants a 15-inch laptop bag. The page shows 15-inch laptop bags and cases, so the practical need is covered. This looks like a strong match.

### Avoid
> The user is seeking a 15-inch laptop bag. The landing page presents laptop bags and cases compatible with 15-inch devices. Therefore, the result satisfies the stated intent.

### Rule
Mix short and medium-length sentences. Do not over-polish every sentence into the same formal rhythm.

---

## 6. Use Certainty and Hedging Deliberately

Be direct when the evidence is clear. Hedge when the judgment depends on interpretation.

### Use direct wording for visible facts
- “The page shows…”
- “The title says…”
- “The product is…”
- “This does not mention…”

### Use mild hedging for interpretation
- “appears to”
- “seems to”
- “likely”
- “could still be useful”
- “may satisfy some users”

### Prefer
> The visible text is a little ambiguous, but it seems to point to a farming-themed game.

### Avoid
> This may perhaps potentially be interpreted as possibly related to a game.

### Rule
Hedging should clarify uncertainty, not weaken the whole comment.

---

## 7. Keep Professional Imperfection

Comments should sound written, not generated. Slightly conversational phrasing is fine when it stays clear and professional.

### Natural
- “not really what the user asked for”
- “still the same type of product”
- “close, but not exact”
- “the page gets part of the query right”

### Over-polished
- “insufficiently responsive to the dominant user objective”
- “maintains category-level relevance while diverging materially”

### Rule
A small amount of everyday phrasing makes comments sound more human. Avoid slang, jokes, and overly casual tone.

---

## 8. Explain the Judgment, Not the Whole Framework

Use the rating framework internally, but do not turn the comment into a policy summary.

### Prefer
> These are related in a broad sense, but they are not interchangeable. Someone looking for a lawn mower is not necessarily looking for a recurring lawn-care service.

### Avoid
> The result preserves an adjacent end-goal while failing the user’s expected solution category, making the substitute structurally weak.

### Rule
State the practical user-facing reason the rating makes sense.

---

## 9. Distinguish “Related” From “Useful”

A result can be connected to the query without being a good answer to it.

### Prefer
> It is in the same general area, but it does not really solve what the user asked for.

### Avoid
> It is related, so it should be considered useful.

### Rule
Do not treat topical overlap as enough by itself. Say whether the result would plausibly help the user.

---

## 10. Avoid Common AI-Sounding Patterns

### Avoid excessive abstract nouns
- alignment
- fulfillment
- utility
- correspondence
- substitution
- differentiation
- optimization

Use them sparingly and only when they are the clearest words.

### Avoid inflated phrasing
- “clearly demonstrates”
- “effectively addresses”
- “substantially fulfills”
- “strongly indicates”
- “notwithstanding”
- “commercially adjacent”

### Avoid mechanical templates
- every comment beginning with “The user is looking for…”
- every comment ending with “Therefore…”
- every comment using three perfectly balanced sentences

### Rule
Repetition makes comments feel generated. Vary the wording while keeping the logic clear.

---

## 11. Prefer Practical User-Perspective Language

When useful, explain how a real user would likely react.

### Prefer
> Someone searching for this exact model probably would not treat a different brand as equally good.

### Prefer
> Some users may still consider this, but it is not as close as the original query suggests.

### Avoid
> A subset of users may regard the offered alternative as sufficiently responsive.

### Rule
Use realistic user reactions instead of abstract audience language.

---

## 12. Better vs. Worse Phrasing Examples

### Retailer mismatch
**Prefer**
> The product matches, but the retailer does not. The user asked for Walmart and this is from Target, so it is useful as an alternative rather than an exact match.

**Avoid**
> The result demonstrates meaningful intent overlap while diverging from the queried merchant specification.

---

### Partial product match
**Prefer**
> This gets part of the query right, but not the full thing. It shows amber cabochons, while the user asked for an amber cabochon silver ring.

**Avoid**
> The page exhibits partial correspondence with the query because it satisfies selected product dimensions while omitting others.

---

### Weak substitute
**Prefer**
> These are related in a broad sense, but they are not interchangeable. Someone looking for a lawn mower is not necessarily looking for a recurring lawn-care service.

**Avoid**
> Despite thematic relatedness, the offering fails to preserve the user’s dominant functional requirement.

---

### Strong match
**Prefer**
> This is exactly what the user asked for: 15-inch laptop bags, with several options shown on the page.

**Avoid**
> The result fully satisfies all explicitly stated user requirements.

---

### Uncertain classification
**Prefer**
> The visible text is a little ambiguous, but it seems to point to a farming-themed game. Since the creative only indirectly signals that it is a game, I rated Yes with some uncertainty.

**Avoid**
> The content may plausibly imply a gaming-related experience, though explicit confirmation is not present, resulting in a cautiously affirmative judgment.

---

## 13. Drafting Checklist

Before finalizing a comment, check:

- Did I state the actual reason for the rating?
- Did I mention the specific match or mismatch, not just say “relevant”?
- Is the wording plain enough to sound natural?
- Did I use certainty where the evidence is clear and hedging only where needed?
- Does the comment explain the practical user impact?
- Could this same comment fit many unrelated tasks? If yes, make it more specific.
- Did I avoid overusing framework vocabulary?
- Does it sound like a person wrote it, rather than a polished generic assistant?

---

## 14. Default Comment Pattern

Use this as a fallback when drafting from scratch:

> The user is looking for [query intent]. The result shows [visible offer]. Since [key overlap] is present but [important difference] remains, I would treat this as [brief rating logic].

### Example
> The user is looking for a specific Nike running shoe model. The result shows Adidas running shoes instead. Since it preserves the broad product need but changes the requested brand and model, it works as a substitute rather than an exact match.
